# Expo Router SSR

This project evaluates whether Expo Server Side Rendering (SSR) and its complementary features (data loaders, server middleware, etc.) are viable for production-level frontend applications. As of Expo SDK 55, SSR is still in alpha, so the goal is to validate real-world feasibility through a simple end-to-end authentication flow.

## Highlights:

- Server-rendered routes (Login, Sign up, Dashboard) — verifiable via curl
- `Expo API Routes` with an in-memory JSON database
- `Expo Router` for navigation and routing
- JWT authentication with refresh tokens using `jose`
- Custom server middleware handling:
  - Request logging
  - Auth guard
  - Redirects
  - Automatic token refresh
- HTTP-only cookies for secure token storage
- SEO support via SSR
- Custom `+html.tsx` setup
- Custom form handling hook
- Structured error handling with custom messages
- Dockerized environment with multistage build and healthcheck
- Environment variables properly isolated (no client leakage)

## How to run the app

- Clone this repository
- Run `yarn` to install dependencies
- On the root folder, run `cp .env.example .env` to setup the env file
- Set a value for `JWT_SECRET` inside `.env`
- run `docker-compose up` to quickstart and access the app at `http://localhost:3000`
- or run `yarn web` to start the bundler and access the app at `http://localhost:8081`

## Architecture decisions

### Expo Server Rendering

Expo SSR was chosen to evaluate it's capabilies on a real world environment. The main reasons are:

- A single project for mobile and web
- File-based routing simplifies structure
- Built-in support for loaders and middleware

The main disadvantages would be that it's still an experimental phase as of SDK 55, so there is no guarantee of avoiding breaking changes or full stability at production level at this moment.

### Monolithic architecture

This app uses Expo API routes instead of a separate backend because it's faster for development and iteration, some logic can be shared between server and client and it's a single development unit. This setup is great for setting up a quick proof of concept like this, even though it's not ideal for large scale systems, since they need more independent flexibilty and scaling.

### JWT Authentication, Refresh Tokens and HTTP-only Cookies

This project uses JWT for authentication with tokens signed by `jose`. This stateless authentication method works great with SSR, and the refresh token strategy provides a great experience for the user. The HTTP-only cookies do not leak to the client, preventing xss attacks - even though it is recommended to setup CSRF protection. Also, cookies are automatically included in requests.

### In memory database

This project uses an JSON in memory database generated by the `src/utils/db.ts` file. The goal here was simplicity and faster setup of the project, avoids extra dependencies but it's not persistent and not recommended for production-level applications.

### Centralized middleware, Data Loaders - and why is the middleware handling redirects

The goal of the middleware on this project is to handle request logging, auth guard, token refresh and redirects. It's a centralized way to handle these features, which improves maintainability and make the request lifecycle more predictable. It holds the extra responsibility of handling redirects because of a limitation of data loaders. According to the current data loaders documentation, a known issue is:

```
Loaders must return JSON-serializable data. Streaming responses are currently not supported. This will be addressed in a future release.
```

This means that if the loader returns a 302, the fetch automatically follows the redirect and attempts to download the JSON from the new URL (e.g., /dashboard). Since /dashboard likely renders an HTML page and not a loader JSON, the parser fails.
Since the goal was to keep redirects on the server side, it was a better option to let the middleware handle it. Data loaders are used only on the dashboard route in order to get data from the cookies before rendering.

### Custom form handler

I created a simple custom form handler for this project that takes care of the values, errors and touched states. I chose it because it avoids adding new packages to the project, and since the forms are really simple, it does a good job handling them.

### Dockerized setup

- Reproducible environment
- One-command startup
- Closer to production setup
